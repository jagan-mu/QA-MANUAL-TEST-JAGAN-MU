# API Testing Questions

**Question 15:** What is the difference between REST APIs and SOAP APIs?
**Answer 15:** <!-- <The primary difference between SOAP and REST is that SOAP is a strict, official protocol governed by the W3C, while REST is an architectural style built around loose guidelines and web standards. SOAP relies exclusively on the XML data format and features built-in compliance policies for high-security applications. Conversely, REST is highly flexible, supporting multiple data formats like JSON, making it lightweight and the dominant choice for modern web and mobile applications . Core Architectural DifferencesSOAP (Simple Object Access Protocol)Operation Oriented: Functions are defined in the payload body (e.g., <CreateUser>) to execute logic.Strict Contract: Relies on a Web Services Description Language (WSDL) file to explicitly define methods and data types.High Security: Uses WS-Security for message-level encryption, making it a favorite for financial services.Heavyweight: Requires rigid XML wrappers (envelopes), causing higher bandwidth overhead.REST (Representational State Transfer)Resource Oriented: Interacts with specific resources exposed via standard URIs.HTTP Methods: Performs CRUD operations directly using standard verbs like GET, POST, PUT, and DELETE.Lightweight Payloads: Prefers JSON, which reduces message size and parsing complexity.Highly Scalable: Because servers do not store client session states, scaling horizontally across servers is simple> -->

**Question 16:** What is the significance of HTTP methods like GET, POST, PUT, and DELETE in REST APIs?
**Answer 16:** <!-- <HTTP methods like GET, POST, PUT, and DELETE act as standard verbs that specify exactly what action to perform on a given server resource. By pairing these methods with resource URIs, ⁠REST APIs map seamlessly to standard CRUD (Create, Read, Update, Delete) database operations, making web communication predictable and uniform across the internet.
Detailed Significance of Each Method
1.GET (Read)Purpose: Retrieves data from a specific endpoint.Safety: It is Safe. It only reads data and never modifies the database.Idempotency: It is Idempotent. Making the same GET request 10 times will always return the exact same resource state.Caching: Browsers and CDN networks automatically cache GET responses to significantly boost performance.2. POST (Create)Purpose: Submits a payload to the server to spin up a new resource.Non-Idempotent: It is not idempotent. If you hit a POST /orders endpoint five times, you will accidentally generate five separate orders in the system.Response: A successful creation returns a 201 Created status along with a Location header pointing to the new item.3. PUT (Update / Replace)Purpose: Updates an existing resource by completely replacing it.Full Payload Rule: You must pass the entire resource representation in the request body. Missing fields are often cleared or set to default values.Idempotency: It is Idempotent. Replacing a specific resource multiple times with identical data leaves the server in the exact same state as the first request.(Note: For partial modifications where you only change one field, use the PATCH method instead).4. DELETE (Delete)Purpose: Erases the resource identified by the unique URI.Idempotency: It is Idempotent. The first request deletes the target resource. Subsequent identical requests will continue to leave the server in that deleted state (though they may return a 404 Not Found error instead of 204 No Content).> -->

**Question 21:** What are status codes (e.g., 200, 400, 500) in an API response?
**Answer 21:** <!-- <An API status code (technically an HTTP response status code) is a three-digit number issued by a server to tell an API client whether a specific network request was successfully completed.These standardized codes act like a quick status update between computers, allowing developers and applications to instantly understand the result of an action without digging through the entire text response.The 5 Standard CategoriesStatus codes are divided into five distinct numerical classes based on their first digit.Code RangeCategoryUniversal Meaning100 – 199InformationalThe server received the request and is still processing it.200 – 299SuccessThe action was successfully received, understood, and accepted.300 – 399RedirectionFurther action or a different URL is needed to complete the request.400 – 499Client ErrorThe request contains bad syntax or cannot be fulfilled by the client.500 – 599Server ErrorThe client did everything right, but the server failed to process it.Most Common API Status Codes🟢 Success (2xx Series)200 OK: The request succeeded, and the server returned the exact data requested.201 Created: The request succeeded and directly resulted in creating a new resource (e.g., a new user profile).204 No Content: The request was successful, but there is no data to send back (common for DELETE actions).🟡 Redirection (3xx Series)301 Moved Permanently: The requested API endpoint has been assigned a new permanent URL.304 Not Modified: Tells the client the data hasn't changed, so it can use its cached copy to save bandwidth.🟠 Client Errors (4xx Series)400 Bad Request: The server cannot process the request due to a client-side error, like sending poorly formatted JSON data.401 Unauthorized: Authentication is required. The client forgot to include an API key or log-in token.403 Forbidden: The server knows who the client is, but they do not have permission to view or modify that specific resource.404 Not Found: The server cannot find the requested resource or endpoint URL.429 Too Many Requests: The client has hit the API rate limit by sending too many requests in a short time.🔴 Server Errors (5xx Series)500 Internal Server Error: A generic "catch-all" error meaning something unexpected broke on the server side.502 Bad Gateway: The server acting as a gateway received an invalid response from the upstream database or application server.503 Service Unavailable: The server is temporarily down for maintenance or overloaded with traffic.> -->
